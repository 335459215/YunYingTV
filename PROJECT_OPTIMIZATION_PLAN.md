## 二、问题识别与分析

### 2.1 冗余代码与重复实现

#### 问题 2.1.1 API 适配器重复实现

**严重程度**: 🟡 中等

**问题描述**:
项目中存在两个 LunaTV API 适配器，职责边界不清：

| 文件 | 描述 | 问题 |
|-----|------|-----|
| `services/apiAdapter.ts` | 提供静态工具方法 (`ApiAdapter` 类) | 功能与 lunaTVAdapter 部分重叠 |
| `services/lunaTVAdapter.ts` | `API` 类的扩展子类 (`LunaTVAdapter`) | 完整实现但与 apiAdapter 功能重复 |

**影响**:
- 代码维护困难，两份代码需要同步更新
- 增加应用包体积
- 开发人员困惑，不知道该用哪个

**建议**:
- 保留 `lunaTVAdapter.ts` 作为完整的 API 扩展类
- 将 `apiAdapter.ts` 中的通用适配逻辑合并到 `api.ts` 或 `lunaTVAdapter.ts`
- 删除 `apiAdapter.ts` 或将其转换为纯类型定义文件

---

#### 问题 2.1.2 M3U 解析文件疑似重复

**严重程度**: 🟢 低

**问题描述**:
`services/` 目录同时存在 `m3u.ts` 和 `m3u8.ts` 两个文件，需要确认是否有重复实现。

**建议**:
- 审查两个文件的具体实现
- 如果功能重叠，合并为一个文件
- 如果是不同格式的解析器，保持但需要清晰的命名

---

#### 问题 2.1.3 平台特定组件重复

**严重程度**: 🟡 中等

**问题描述**:
`VideoCard` 组件存在 4 个平台特定版本：
- `VideoCard.tsx` - 基础版本
- `VideoCard.mobile.tsx` - 移动端
- `VideoCard.tablet.tsx` - 平板
- `VideoCard.tv.tsx` - TV 端

**观察**:
这是合理的响应式设计模式，但需要确保：
- 基础版本包含尽可能多的共享逻辑
- 平台特定版本仅包含平台差异代码
- 避免跨平台版本的代码复制

---

### 2.2 日志系统不一致

#### 问题 2.2.1 console.* 调用未替换

**严重程度**: 🟡 中等

**问题描述**:
项目中仍有 **32 处** `console.*` 调用分散在 8 个文件中：

| 文件 | console 调用数量 |
|-----|------------------|
| `utils/PerformanceMonitor.ts` | 2 |
| `services/lunaTVAdapter.ts` | 10 |
| `hooks/useInteractionOptimize.ts` | 1 |
| `services/apiAdapter.ts` | 1 |
| `hooks/useVideoHandlers.ts` | 11 |
| `utils/Logger.ts` | 4 |
| `stores/updateStore.ts` | 2 |
| `services/m3u.ts` | 1 |

**影响**:
- 生产环境日志无法控制
- 无法利用 Logger 的分级日志功能
- 性能影响（console.log 在生产环境仍会执行）

**建议**:
- 将所有 `console.*` 替换为 `Logger.*`
- 在开发环境保留详细日志
- 生产环境通过 Logger 级别控制日志输出

### 2.4 架构与设计问题

#### 问题 2.4.1 缓存管理潜在内存泄漏

**严重程度**: 🟡 中等

**问题描述**:
`homeStore.ts` 中使用模块级别的 `dataCache` Map：

```typescript
const dataCache = new Map<string, CacheItem>();
```

**问题**:
- 缓存在应用生命周期内永久存储
- 没有主动清理机制（仅在写入时清理过期项）
- 如果缓存数据量大，可能导致内存压力

**建议**:
- 考虑使用 WeakMap 或添加 LRU 策略
- 添加 `clearCache()` 方法供手动清理
- 考虑使用 React Native 的 MMKV 替代内存缓存

---

#### 问题 2.4.2 Store 访问模式不一致

**严重程度**: 🟢 低

**问题描述**:
- 部分地方使用 `useStore()` hook
- 部分地方使用 `Store.getState()` 静态访问

**示例**:
```typescript
// hook 方式
const { isLoggedIn, logout } = useAuthStore();

// 静态方式
const { autoContinuePlayback } = useSettingsStore.getState();
```

**建议**:
- 统一 Store 访问模式
- 组件内使用 hook 方式
- 非组件上下文使用静态方式
- 添加 ESLint 规则区分使用场景

---

#### 问题 2.4.3 类型安全不足

**严重程度**: 🟡 中等

**问题描述**:
`apiAdapter.ts` 和 `lunaTVAdapter.ts` 中使用大量 `any` 类型：

```typescript
static adaptSearchResult(data: any, source: string): SearchResult[]
static adaptVideoDetail(data: any, source: string): VideoDetail | null
```

**建议**:
- 使用泛型或联合类型替代 `any`
- 创建明确的接口定义

---

### 2.5 安全与配置问题

#### 问题 2.5.1 Cleartext Traffic 配置

**严重程度**: 🟢 低 (开发环境正常)

**问题描述**:
`app.json` 中配置 `usesCleartextTraffic: true`

```json
"android": {
  "usesCleartextTraffic": true
}
```

**说明**:
- 允许 HTTP 明文传输
- 对于连接本地开发服务器是必需的
- 生产环境应限制为 HTTPS

**建议**:
- 添加开发/生产环境区分配置
- 在文档中明确说明此配置的用途
- 考虑使用网络安全配置限制仅本地开发服务器

### 2.6 测试覆盖不足

#### 问题 2.6.1 单元测试覆盖范围有限

**问题描述**:

- 添加集成测试框架
- 设置 CI 测试覆盖率门槛
